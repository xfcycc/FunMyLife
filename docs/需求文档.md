# Life Manager 需求文档

## 1. 文档定位

本文是 Life Manager 当前阶段的产品主需求文档，用于统一产品方向、核心模型、导航结构、页面职责、方案机制、功能块规则、MVP 范围和后续演进。

本文需要与 [Life Manager 项目概念与方案设计.md](./Life%20Manager%20项目概念与方案设计.md) 保持一致。具体项目级设计先以 [Life Manager 无限暖暖项目设计草案.md](./Life%20Manager%20无限暖暖项目设计草案.md) 为讨论样板；进入前端 mock 和页面闭环时参考 [Life Manager 无限暖暖数据模型与页面闭环设计.md](./Life%20Manager%20无限暖暖数据模型与页面闭环设计.md)。后续讨论新项目、新方案、新页面和 AI 协作时，优先使用“项目、类型、周期属性、能力、功能块、功能块实例、方案、用户数据”这套语言。

本文不展开数据库表结构、后端接口字段、权限模型、低代码搭建器实现和社区方案市场实现。第一阶段重点是把产品概念、页面职责和可演示闭环先稳定下来。

## 2. 产品概述

Life Manager 是一个面向个人使用的生活项目管理系统。它不是普通待办工具，也不是传统后台管理系统，而是帮助用户把生活中长期存在、阶段推进、需要持续维护的事情组织成项目。

用户可以把一直玩的游戏、明年的旅行、长期学习计划、家庭保险、人际关系维护、订阅续费、健康管理等生活对象创建成项目。每个项目可以聚合任务、清单、倒计时、提醒、资料、预算、联系人、笔记、图册、AI 建议、外部消息和最近动态。

当前项目底座为 Vue 3 / TypeScript / Vite / Soybean Admin 前端，以及 RuoYi-Vue-Plus / Spring Boot 后端。第一阶段接受现有 Vue 底座，复用登录、权限、OSS、SSE、菜单和后端基础能力，但主体验需要弱化后台感，围绕 Life Manager 的项目体验展开。

## 3. 核心产品模型

Life Manager 的核心模型是：

```text
项目 = 类型 + 周期属性 + 方案 + 功能块实例 + 用户数据
功能块 = 一个或多个能力的产品化组合
功能块实例 = 功能块 + 项目内配置
```

项目是用户真正创建和使用的生活对象，例如“我的无限暖暖”“日本旅行 2026”“家庭保险”“OpenAI 学习计划”。项目承载真实数据，并随着使用不断沉淀内容。

类型是项目所属的大类，例如游戏、旅行、保险、人际关系、学习、健康。类型用于帮助用户理解项目、筛选项目、选择方案，也帮助 AI 判断项目语境。类型不直接决定完整页面和字段。

周期属性说明项目如何随时间推进。长期经营型项目没有明确结束点，适合一直玩的游戏、长期学习、健康管理和人际关系维护；阶段计划型项目有明确目标或完成节点，适合旅行、考试、搬家、一次活动、理赔等。

方案是某个具体场景的默认搭法，例如无限暖暖方案、原神方案、三角洲行动方案、日本自由行方案、家庭保险方案、朋友关系维护方案。方案决定默认关注点、默认功能块、详情页展示顺序、功能块 Tab 配置和项目管理页配置项。

能力是底层可复用机制，例如目标系统、倒计时、资料资产、媒体记录、时间轴回顾、摘要提醒、AI 总结。功能块是一个或多个能力的产品化组合，例如任务、活动与版本、资料、图册、AI 建议。同一个功能块在不同方案中可以使用不同显示名称、不同信息项和不同展示方式。

功能块被启用到某个项目后，会形成一个“项目功能块实例”。实例不只是一个 Tab，还应包含这个项目里的显示名称、字段扩展、摘要规则、提醒规则、时间轴记录规则和权限策略。比如同样是任务功能块，无限暖暖可以配置为“日常任务”，并决定概览显示今日任务、本周任务、活动任务还是只显示异常未完成项。

这套模型的核心价值是：以后新增项目时，不再从“这个页面有哪些字段”开始，而是先判断它属于什么类型、如何随时间推进、适合使用哪个方案、需要哪些功能块。

## 4. 产品目标

第一阶段的目标是把当前系统从“后台底座”改造成“个人生活项目管理产品雏形”。

产品需要让用户能用项目组织生活对象。长期经营型项目要能持续记录任务、动态、资料、复盘和趋势；阶段计划型项目要能围绕目标推进准备、执行、完成和归档。

产品需要让用户能从方案开始创建项目。用户不应该面对一组空白字段，而应该从“无限暖暖方案”“日本自由行方案”“家庭保险方案”这类默认搭法开始，再按自己的情况调整。

产品需要让功能块可以被复用。任务、倒计时、资料、图册、联系人、预算等能力不应该只属于某一个页面，而应该能被不同项目方案组合、改名、排序和配置。

产品需要让 AI 能理解这套结构。AI 应能帮助用户设计新项目、推荐方案、判断功能块、总结项目状态、提取待确认内容和给出风险提示，但第一阶段不应自动修改关键资料。

产品需要支持外部渠道接入。飞书、OpenClaw、Webhook、MCP 等能力可以用于提醒、查询、候选内容收集和自动化，但所有敏感写入和关键资料变更都必须经过用户确认。

## 5. 设计原则

### 5.1 项目优先

项目是 Life Manager 的第一组织方式。首页、项目列表、项目详情和项目管理页都应围绕项目展开。

功能块仍然重要，但它们不应该默认平铺成一堆一级导航。日程、清单、资料、图册、提醒更像可复用能力，应该优先出现在项目详情页、首页摘要、项目管理配置、跨项目搜索和后续功能视图中。

### 5.2 方案优先于字段

新项目设计不从页面字段开始，而从方案开始。一个方案必须回答：适用场景是什么，默认周期属性是什么，用户主要关注什么，默认启用哪些功能块，哪些功能块进入项目导航，管理页配置什么。

例如同样是游戏类型，无限暖暖关注日常、活动、搭配、素材、拍照和攻略；原神关注树脂、每日委托、深渊、角色养成、抽卡计划和版本活动；三角洲行动关注赛季、通行证、武器配置、战绩复盘和组队。它们不应该共用一个固定“游戏页面”，而应该是同一类型下的不同方案。

### 5.3 功能块复用但不强行抽象

多个方案都需要的能力，应沉淀成通用功能块。只有某个方案需要的特殊内容，可以先留在该方案内部，不急着抽象。

功能块可以被重命名、合并、隐藏或拆分。例如“倒计时”在无限暖暖里可以叫活动倒计时，在原神里可以叫卡池倒计时，在旅行里可以叫出发倒计时，在保险里可以叫缴费倒计时。

### 5.4 日常数据和规则配置分离

功能块 Tab 负责日常数据，项目管理页负责结构和规则。

新增一个账号资产、新增一个倒计时、完成一条任务、上传一张照片、记录一篇笔记，应该发生在对应功能块 Tab 中。启用哪些功能块、默认提醒时间、资料分组、敏感策略、同步范围、AI 规则和外部渠道配置，应该发生在项目管理页中。

项目管理页不能只有一组笼统配置。每个已启用的功能块都需要有自己的配置入口，用来控制字段、分组、摘要、提醒、时间轴记录和 AI 行为。功能块 Tab 解决“我现在要维护什么数据”，功能块配置解决“这个功能块在本项目里如何工作”。

### 5.5 敏感内容默认保守

系统会保存账号、订单、保险、证件、联系人、聊天内容和外部渠道 Token。敏感信息默认不在首页、项目卡片和普通列表中明文展示。

AI 和外部渠道可以做候选提取、摘要、风险提示和建议，但不能无确认写入关键资料、联系人、预算、证件、账号和保险内容。

## 6. 信息架构

### 6.1 一级导航

第一阶段一级导航应围绕用户心智和核心流程组织，不直接平铺所有功能块。

```text
工作台
项目
方案
待确认
AI 助手
设置
```

工作台是每日总览入口，对应当前 `/life/home`。它展示今天要关注什么、哪些项目有变化、哪些事项需要处理。

项目是项目列表和项目管理入口，对应当前 `/life/projects`。它用于查看、筛选、创建、归档和进入具体项目。

方案是项目方案库入口。第一阶段可以是轻量页面或创建项目时的选择入口，用于查看已有方案、理解方案结构、从方案创建项目，后续承接朋友和社区贡献的方案。

待确认是统一收件箱。外部消息、AI 提取结果、导入内容、风险提示和无法自动归属的候选内容都应先进入这里，再由用户确认写入项目。

AI 助手用于跨项目问答、项目设计建议、今日建议、总结和辅助整理。

设置承接账号、系统、外部渠道、开发者能力和底座管理入口。系统管理相关页面可以放在设置或开发者区域，不作为 Life Manager 主体验暴露。

### 6.2 功能块入口

日程、清单、资料、图册、提醒、联系人、预算等功能块第一阶段不建议全部作为一级导航。

它们可以出现在项目详情页内、首页摘要组件内、搜索结果内、项目管理配置内、待确认详情内，以及后续需要时的独立功能视图。只有当某个功能块存在强烈跨项目管理需求时，才考虑升级为二级或一级入口。

### 6.3 项目内部导航

项目内部导航由方案决定，不是固定结构。

项目内部通常包含概览、若干功能块 Tab、AI、管理。不同方案可以使用不同名称，但底层功能块尽量复用。

例如无限暖暖可以是“概览 / 日常 / 倒计时 / 账号资产 / 笔记 / 图册 / AI”；日本旅行可以是“概览 / 行程 / 清单 / 资料 / 预算 / 图册 / AI”；保险可以是“概览 / 保单资料 / 缴费提醒 / 联系人 / 附件 / AI”；人际关系可以是“概览 / 互动记录 / 重要日期 / 偏好资料 / 礼物清单 / AI”。

### 6.4 创建项目流程

创建项目应优先从方案开始。

```text
选择类型 -> 选择方案 -> 确认周期属性 -> 预览功能块 -> 创建项目 -> 进入项目管理页补充配置
```

如果用户无法判断类型或方案，AI 可以根据自然语言描述推荐类型、周期属性、方案和功能块。

## 7. 页面需求

### 7.1 工作台 `/life/home`

工作台是用户每天打开 Life Manager 的第一屏。它不是完整项目列表，也不是功能块入口集合，而是帮助用户快速判断今天应该关注什么。

工作台应以项目为中心，同时展示跨项目摘要。建议包含置顶项目、需关注项目、今日跨项目事项、临近倒计时、即将到期资料、待确认内容、AI 今日建议和实时状态。

工作台里的功能块只能作为摘要出现。例如倒计时提醒展示最近临近的几个节点，资料提醒展示即将到期或待补全资料，待确认展示 AI 或外部渠道提取出的候选内容。用户点击后应进入对应项目、对应功能块 Tab 或待确认详情，而不是默认进入孤立功能页。

长期经营型项目在工作台中更适合展示今日任务、周期任务、趋势变化和最近动态。阶段计划型项目更适合展示倒计时、里程碑、准备清单和风险提醒。

工作台不承担完整项目管理职责。完整筛选、批量管理、归档和恢复应在项目页完成。

### 7.2 项目页 `/life/projects`

项目页用于统一查看、筛选、创建和管理所有项目。

项目列表需要支持按类型、周期属性、状态、方案、是否归档进行筛选，也需要支持搜索、排序、视图切换、新建项目、编辑项目、归档项目和恢复项目。

项目卡片需要展示项目名称、类型、周期属性、方案、状态、最近动态、下一个提醒、已启用功能块、AI 整理状态和项目风险提示。

项目页还应承接从方案创建项目的入口。第一阶段至少需要能表达无限暖暖方案、日本旅行方案，以及原神、三角洲行动、保险、人际关系等方案草案。

### 7.3 项目详情页

项目详情页是用户日常使用某个项目的主要入口。它由方案和功能块决定，不应该是固定页面结构。

详情页应包含项目 Hero、关键状态、概览、主要功能块 Tab、最近动态和 AI 建议。长期经营型项目应突出周期任务、最近动态、阶段目标和复盘；阶段计划型项目应突出倒计时、里程碑、准备事项、执行安排和风险提醒。

项目详情页的目标是让用户进入项目后，马上知道这个项目当前最需要关注什么，并能快速进入任务、资料、图册、笔记、联系人、预算或 AI 建议等功能块。

### 7.4 项目概览

概览是项目详情页的默认入口，负责展示项目重点状态和各功能块摘要。

概览不承担完整数据管理职责，而是帮助用户快速判断当前最重要的事项，并提供进入对应功能块的快捷入口。

例如无限暖暖概览可以展示今日任务进度、2 个临近活动倒计时、3 个账号资产状态、素材变化、图册预览和 AI 建议。日本旅行概览可以展示出发倒计时、近期行程、行前清单进度、关键资料、预算状态和天气提醒。

概览卡片里的“查看全部”“管理”“新增”应进入对应功能块 Tab 或项目管理页，而不是在概览里展开完整管理体验。

概览展示内容必须可配置。每个功能块实例需要声明自己在概览中的摘要规则，例如显示今日任务、本周任务、临近到期、异常状态、最近新增、最近完成、核心指标或完全不显示。对于无限暖暖这类长期经营项目，任务摘要可以配置为“优先显示今日未完成任务，同时显示本周进度和活动任务提醒”，而不是固定只显示今日任务。

### 7.5 功能块 Tab

功能块 Tab 负责完整数据查看和日常数据维护。

用户在功能块 Tab 中查看该功能块的全部数据，并完成新增、编辑、删除、完成、恢复、打卡、归档、查看详情等高频操作。

例如账号资产 Tab 应展示全部账号资产，并支持新增账号、编辑状态、查看详情、停用或归档。倒计时 Tab 应展示全部活动、版本节点、卡池、赛季或出发节点，并支持新增、编辑、设置提醒、完成或归档。图册 Tab 应展示完整相册和最近照片，并支持上传、查看详情、关联笔记或归档。

这意味着项目内的数据录入和日常维护主要发生在功能块 Tab，而不是项目管理页。

### 7.6 项目管理页

项目管理页负责项目结构和规则配置，不负责逐条承接所有日常数据。

它管理的是更上层的规则，例如项目基础信息、启用或停用哪些功能块、功能块排序、功能块显示名称、资料分组、敏感字段策略、默认提醒规则、同步规则、AI 规则、外部渠道和方案继承关系。

项目管理页需要提供功能块实例配置。用户进入某个已启用功能块后，可以配置该功能块在本项目里的字段、分组、筛选、摘要卡片、提醒策略、时间轴记录策略、AI 建议范围和外部渠道行为。

例如无限暖暖的“日常任务”配置应至少能控制：任务分组是否包含今日、周常、活动；每个分组的重置时间；概览卡片显示今日任务、本周任务还是活动任务；任务完成后是否写入时间轴；时间轴记录粒度是逐条记录还是每日汇总；长期回顾里是否统计连续完成天数和遗漏次数。

项目管理页需要区分“方案默认配置”和“用户个性化配置”。后续如果方案升级，系统需要知道哪些内容来自方案，哪些内容是用户覆盖。

项目管理页成功标准是：用户可以完成项目初始化和后续调整；修改功能块启停、提醒规则、资料分组、AI 设置和同步规则后，项目详情页和工作台能正确反映变化。

### 7.7 功能块与 Tab 的关系

功能块和项目 Tab 默认接近一一对应，但不能强绑定。

重要且高频的功能块通常适合成为独立 Tab，例如任务、倒计时、资料、图册、笔记、AI。轻量功能块可以只出现在概览或合并进其他 Tab。复杂功能块可以拆成多个子视图。某些功能块也可以只存在于待确认收件箱或项目管理页，不一定进入项目导航。

方案需要声明每个功能块的呈现方式：

```text
哪些功能块进入项目导航
哪些功能块只在概览展示摘要
哪些功能块合并到其他 Tab
哪些功能块只在管理页配置
哪些功能块由待确认收件箱承接
```

### 7.8 项目时间轴

长期经营型项目必须有时间轴能力。时间轴用于回答“这个项目过去发生了什么”，不是简单的最近动态列表。

时间轴可以接收来自功能块的事件，例如完成任务、跳过任务、创建倒计时、活动结束、上传图册、补充资料、修改账号安全状态、AI 生成周总结、外部渠道写入候选内容。每条事件需要保留发生时间、来源功能块、事件类型、摘要、关联对象和是否敏感。

时间轴展示可以出现在项目概览、项目详情的独立 Tab 或 AI 总结中。长期项目应支持按日、按周、按月回顾；阶段计划型项目应支持按阶段回顾。

时间轴记录不应由系统硬编码。每个功能块实例都需要配置哪些操作写入时间轴、写入为逐条事件还是聚合摘要、是否在概览展示、是否允许 AI 使用。

## 8. 重点方案需求

### 8.1 无限暖暖方案

无限暖暖是游戏类型的长期经营型项目，适合作为游戏方案的第一批样板。

它关注日常任务、周常目标、活动倒计时、版本直播、账号资料、素材收集、截图图册、攻略笔记、长期时间轴、AI 项目建议和最近动态。

详情页应让用户马上知道今天游戏里还有什么没做，本周目标推进到哪里，活动还有多久结束，账号资料是否安全，素材和图册有哪些新变化，以及最近几天项目发生了什么。管理页重点配置任务模板、功能块摘要规则、时间轴记录规则、提醒规则、资料分组、图册同步、AI 设置和外部通知。

### 8.2 日本旅行方案

日本旅行 2026 是旅行类型的阶段计划型项目，适合作为旅行方案的第一批样板。

它关注出发倒计时、近期行程、行前清单、机票酒店资料、保险和证件资料、预算概览、天气地点、同行人员、图册预览、旅行笔记、AI 行前建议和最近动态。

旅行项目需要支持阶段变化。出发前重点展示准备事项和风险提醒；旅行中重点展示今日行程、资料入口和记录入口；旅行后重点展示图册、日记、消费记录和 AI 总结。

### 8.3 方案草案

第一阶段还应整理若干方案草案，用于验证模型扩展性。

原神方案用于验证同一游戏类型下的差异化关注点，例如树脂、每日委托、深渊、角色养成、抽卡计划和版本活动。

三角洲行动方案用于验证赛季、通行证、武器配置、战绩复盘和组队联系人等偏竞技游戏场景。

家庭保险方案用于验证敏感资料、缴费提醒、保障范围、受益人、联系人和附件管理。

朋友关系维护方案用于验证人际关系类长期经营项目，例如重要日期、最近互动、偏好、礼物、共同经历和待联系提醒。

## 9. 核心能力需求

### 9.1 项目能力

项目需要支持创建、编辑、归档、恢复、搜索、筛选、封面、状态、周期属性、方案来源、已启用功能块、项目内部导航、项目内容聚合和项目 AI 总结。

项目状态至少包括进行中、需关注、已停用、已归档。阶段计划型项目还需要表达准备中、进行中、已完成等阶段变化。

### 9.2 方案能力

方案用于指导项目创建和页面组织。

方案需要包含所属类型、默认周期属性、适用场景、主要关注点、底层能力清单、默认功能块、功能块显示名称、功能块实例配置、详情页展示顺序、管理页配置项和示例数据。

方案还需要声明每个功能块在项目内的呈现方式。一个功能块可以进入项目导航成为独立 Tab，也可以只在概览展示摘要，可以合并到其他 Tab，也可以只在管理页配置或由待确认收件箱承接。

第一阶段方案可以先以文档和 mock 配置形式存在，不急于做可视化方案编辑器。

### 9.3 能力、功能块与功能块实例

能力用于沉淀底层可复用机制，功能块用于把一个或多个能力组织成用户可见、可操作的入口。第一阶段建议优先梳理目标系统、提醒、倒计时、资料资产、预算、联系人、笔记、媒体记录、时间轴、AI 建议、外部消息等能力，并通过任务、资料、图册、AI 等功能块落地。

功能块应支持在不同方案中重命名和调整展示位置。多个方案都需要的底层机制，应优先抽成能力；用户可见、可进入、可操作的组合，应作为功能块；只有单个方案需要的特殊内容，可以先作为方案内部配置或项目内自定义功能块实例。

功能块应区分数据维护能力和规则配置能力。日常数据维护发生在功能块 Tab 内，例如新增账号资产、新增倒计时、完成任务、上传图册、记录笔记；规则配置发生在项目管理页内，例如默认提醒时间、资料分组、敏感策略、同步范围、AI 生成规则。

每个项目功能块实例至少需要具备四类配置：字段配置、展示配置、行为配置和记录配置。字段配置决定这类数据在本项目里多哪些字段；展示配置决定是否进入项目导航、概览显示什么、排序如何；行为配置决定提醒、重置、状态流转和外部同步；记录配置决定哪些操作写入时间轴，以及写入为明细还是汇总。

### 9.4 提醒与倒计时

提醒和倒计时是系统基础能力。

系统需要支持一次性提醒、每日提醒、每周提醒、到期前提醒、自定义周期提醒、项目内提醒、首页提醒和外部渠道提醒。

典型场景包括会员到期、保险缴费、游戏活动结束、旅行出发倒计时、学习复习提醒和重要联系人纪念日。

### 9.5 资料与资产

资料与资产用于管理账号、链接、订单、保险、证件、会员服务、附件和重要凭证。

系统需要支持普通资料、敏感资料、附件、链接、到期提醒、项目关联和安全查看。敏感资料默认不明文展示，需要明确入口和安全提示。

### 9.6 记录、笔记与图册

记录、笔记与图册用于长期沉淀生活内容。

系统需要支持项目关联、标签、摘要、图册预览、附件上传、OSS 同步、上传状态提醒和 AI 整理。旅行照片、游戏截图、保单文件、订单截图都应可以归属到项目。

### 9.7 时间轴与回顾

时间轴用于沉淀项目历史。长期经营型项目需要能回顾每天、每周、每月发生了什么；阶段计划型项目需要能回顾每个阶段做了什么、遗漏了什么、最终结果如何。

时间轴事件可以由用户手动创建，也可以由功能块自动产生。自动产生的事件需要受功能块实例配置控制，避免把所有操作都写成噪音。

时间轴需要支持筛选来源功能块、事件类型、时间范围和敏感级别。AI 可以基于时间轴生成日报、周报、阶段复盘和项目总结，但敏感内容默认不展开。

### 9.8 待确认收件箱

待确认收件箱用于承接不能直接写入项目的数据。

外部消息、AI 提取结果、导入内容、OCR 识别结果、聊天内容中的候选待办和风险提示，都应先进入待确认。用户确认后，系统再写入对应项目和功能块。

待确认内容需要支持归属项目、选择功能块、编辑内容、确认写入、忽略、稍后处理和查看来源。

### 9.9 AI 助手

AI 助手需要围绕项目工作。

AI 需要支持今日建议、项目总结、日记摘要、学习复盘、笔记整理、待办提取、旅行建议、游戏任务建议、资料完整性检查、到期事项提醒、方案设计建议和功能块推荐。

AI 的输出应该作为建议或候选内容。涉及关键资料、预算、证件、账号、联系人和外部消息写入时，必须用户确认。

### 9.10 外部渠道与 MCP

外部渠道用于提醒、候选内容收集和自动化。

第一版建议优先支持飞书提醒、OpenClaw 和 Webhook。后续可以扩展邮件、日历、企业微信、手机推送。

MCP 是后续能力，不一定放在第一版完成。建议支持查询今日事项、查询项目概览、查询即将到期、创建提醒、创建笔记和总结项目。MCP 默认不返回敏感信息，涉及账号密码、证件和联系人时需要明确授权。

## 11. 页面优先级

第一优先级是工作台、项目列表、项目详情、项目管理配置，以及无限暖暖和日本旅行两个样板项目。

第二优先级是方案库、待确认收件箱、AI 助手和跨项目搜索。它们承接项目创建、外部消息、AI 候选内容和跨项目查询。

第三优先级才是提醒、资料、图册、笔记等功能块的独立功能视图。它们需要能与项目关联，但第一阶段不必全部做成独立复杂页面。

设置、外部渠道、MCP 管理、数据导入导出和方案管理后台可以随能力成熟逐步补齐。

## 12. 体验要求

工作台要轻量、清晰、有生活感，不要像复杂企业后台。用户打开后应能快速知道今天应该关注哪些项目和事项。

项目页要让用户感到一个生活对象被完整收纳和持续照看。用户进入项目后，应能马上知道当前最该关注什么。

概览要负责看重点，不要塞入完整管理能力。功能块 Tab 要负责日常数据维护。项目管理页要负责结构和规则配置。

项目创建和配置要围绕方案展开，不要让用户从空白字段开始填。对于不确定的用户，AI 可以帮助推荐方案和功能块。

管理页要实用，不要为了配置而配置。默认方案应承担大部分初始化工作，用户只需要调整少量个性化内容。

AI 建议要短、具体、可执行。敏感信息默认收起，外部写入默认待确认。

页面要适合长期使用，避免过度花哨。视觉上应保持 Life Manager 现有页面风格一致。

## 13. 验收标准

工作台验收：用户能看到置顶项目、需关注项目、今日跨项目事项、临近倒计时、待确认内容、AI 建议和实时状态，并能快速进入对应项目或功能块。

项目列表验收：用户能创建、筛选、搜索、归档项目，能看到每个项目的类型、周期属性、方案、近期状态、下一个提醒和已启用功能块。

项目详情验收：概览能按功能块摘要规则展示重点内容；功能块 Tab 能查看和维护完整数据；项目管理页能进入每个功能块实例配置结构、摘要、提醒和记录规则，三者职责不混乱。

方案模型验收：无限暖暖和日本旅行能被解释为两个方案样板；原神、三角洲行动、保险、人际关系能写出方案草案；新项目设计不再从页面字段开始。

无限暖暖项目验收：用户能看到游戏日常、周常、活动倒计时、版本提醒、资料、笔记、图册、长期时间轴和 AI 建议，能在对应 Tab 维护日常数据，并在管理页配置任务模板、概览显示规则、时间轴记录规则、提醒规则、图册同步和 AI 设置。

日本旅行项目验收：用户能看到出发倒计时、行程、清单、资料、预算、图册、笔记和 AI 建议，能在对应 Tab 维护行程、清单和资料，并在管理页配置同行人员、清单模板、资料分组、图册同步和 AI 设置。

安全验收：敏感内容不会在列表和首页直接明文展示。外部渠道和 AI 调用不会默认暴露账号、证件、保单、联系人等敏感信息。外部消息和 AI 提取结果默认进入待确认流程。

## 14. 后续演进方向

后续可以逐步增强移动端适配、手机推送、浏览器插件、自动识别图片地点和时间、自动生成旅行回忆、个人年度报告、项目方案市场、完整 MCP 工具集、本地模型或多 AI 服务商支持。

当方案数量足够多时，可以考虑开放方案贡献机制，让朋友或社区基于统一协议贡献项目方案。方案贡献不应该直接修改页面代码，而应优先提供类型、周期属性、关注点、默认功能块、展示顺序、配置项和示例数据。
