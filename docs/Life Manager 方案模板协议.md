# Life Manager 方案模板协议

## 1. 文档定位

本文定义 Life Manager 的“方案模板协议”。

它不是数据库表结构，也不是低代码编辑器规格。它是一套设计和描述项目方案的共同格式，用来保证后续新增无限暖暖、原神、日本旅行、家庭保险、人际关系维护等项目时，不再从“每个页面有哪些字段”开始，而是先用同一套语言描述方案。

在产品语境里，统一使用“方案”。如果后续对外说“模板”“预设”“社区方案”，都先视为方案的不同呈现方式。

方案模板协议要服务三类对象。

第一类是人。产品设计、页面设计和朋友/社区贡献方案时，都能看懂一个方案到底解决什么问题。

第二类是系统。前端可以根据方案生成项目导航、概览摘要、管理配置和 mock 数据；后端可以据此设计项目、功能块实例、规则和用户数据关系。

第三类是 AI。用户以后可以问 AI：“帮我设计一个原神项目方案”“这个功能应该是能力、功能块还是字段”“日本旅行方案需要哪些默认配置”。AI 应按本文协议输出，而不是直接罗列页面字段。

## 2. 协议原则

方案不是页面模板。页面只是方案的展示结果。一个方案必须先回答用户在什么场景下使用、长期还是阶段推进、真正关注什么、底层需要哪些能力、哪些能力要组合成可见功能块。

方案不是用户数据。方案可以包含示例数据，但示例数据只用于演示和初始化，不应混入真实项目数据。用户基于方案创建项目后，产生的任务、照片、保单、联系人、时间轴记录都属于用户数据。

方案要允许个性化覆盖。用户创建项目后，可以改功能块显示名称、导航顺序、概览规则、提醒规则、时间轴记录规则和字段配置。方案升级时，系统需要知道哪些是方案默认值，哪些是用户覆盖值。

方案要克制。不是所有能力都要变成 tab，也不是所有字段都要暴露在首页。概览看重点，功能块 tab 管数据，项目管理页管规则，时间轴管回顾。

## 3. 一份方案必须回答的问题

一份合格的方案至少要回答这些问题：

| 问题 | 说明 |
| --- | --- |
| 这是什么方案 | 方案名称、所属类型、适用场景 |
| 适合谁用 | 目标用户、使用频率、默认复杂度 |
| 如何随时间推进 | 长期经营型、阶段计划型，或两者混合 |
| 用户真正关注什么 | 关注点列表，不是字段列表 |
| 需要哪些底层能力 | 目标、倒计时、资料、图册、预算、联系人、时间轴等 |
| 暴露哪些功能块 | 用户能进入、查看、管理的入口 |
| 功能块如何实例化 | 显示名、排序、字段、提醒、归档、时间轴、AI 规则 |
| 概览展示什么 | 由摘要规则生成，不直接堆全部数据 |
| 管理页配置什么 | 项目基础配置和功能块实例配置 |
| 如何初始化项目 | 默认值、示例数据、创建流程 |
| 有哪些安全边界 | 敏感字段、AI 可读范围、外部写入确认 |

如果一个方案无法回答这些问题，就还不能进入页面实现。

## 4. 协议结构

方案模板由十二个部分组成。

```text
方案基础信息
适用范围
周期属性
关注点
能力清单
功能块实例
项目导航
概览摘要规则
时间轴规则
项目管理配置
示例数据
安全与 AI 边界
```

产品设计时优先使用中文概念。代码实现时可以使用英文 key，但英文 key 只服务实现，不替代中文语义。

## 5. 方案基础信息

方案基础信息用于识别这个方案。

| 中文概念 | 推荐 key | 说明 |
| --- | --- | --- |
| 方案 ID | `schemeId` | 稳定唯一值，例如 `game.infinity-nikki` |
| 方案名称 | `schemeName` | 用户看到的名称，例如“无限暖暖方案” |
| 所属类型 | `type` | 游戏、旅行、保险、人际关系等 |
| 默认周期属性 | `cycleType` | 长期经营型、阶段计划型、混合型 |
| 方案版本 | `version` | 用于后续方案升级 |
| 方案状态 | `status` | 草案、可用、归档 |
| 方案来源 | `source` | 官方、用户自定义、社区贡献、AI 生成 |
| 一句话说明 | `summary` | 帮用户判断是否适合自己 |

这里的类型不能过细。“游戏”是类型，“无限暖暖”和“原神”是方案。“旅行”是类型，“日本自由行”是方案。

## 6. 适用范围

适用范围描述谁会用这个方案、什么时候用、默认复杂度多高。

建议包含：

| 中文概念 | 推荐 key | 说明 |
| --- | --- | --- |
| 目标用户 | `targetUser` | 例如长期玩无限暖暖的玩家、第一次去日本自由行的人 |
| 使用频率 | `usageFrequency` | 每日、每周、出行前集中使用、偶尔维护 |
| 默认复杂度 | `complexity` | 轻量、标准、进阶 |
| 不适用场景 | `notFor` | 避免方案被误用 |
| 创建时需要用户补充 | `requiredOnCreate` | 项目名、出发日期、服务器、预算等 |

适用范围要写清楚，是为了让用户和 AI 知道什么时候该选这个方案，什么时候该另建方案。

## 7. 周期属性

周期属性决定方案如何处理时间。

长期经营型项目更关注持续记录、周期目标、阶段复盘和历史沉淀。无限暖暖、原神、长期学习、人际关系维护都属于这一类。

阶段计划型项目更关注目标日期、准备进度、执行计划、风险提醒和完成归档。日本旅行、搬家、考试、理赔都属于这一类。

混合型项目需要说明主周期和内部阶段。例如保险项目整体是长期经营型，但一次理赔是阶段计划型；游戏是长期经营型，但一个版本活动是阶段计划型。

协议里建议这样写：

```text
周期属性：长期经营型
主时间结构：版本、活动、每日重置、每周重置
归档方式：版本归档、活动归档、周回顾
时间轴视角：按天、按周、按版本
```

或者：

```text
周期属性：阶段计划型
主时间结构：出发日期、行程日期、准备期、旅行中、旅行后
归档方式：旅行完成后归档为回忆和总结
时间轴视角：按准备阶段、按日期、按地点
```

## 8. 关注点

关注点是方案设计的核心。关注点回答“用户真正想盯住什么”。

关注点不是字段。比如原神方案的关注点不是“角色名、等级、武器、圣遗物这些字段”，而是“体力不要溢出、深渊不要忘、角色养成有计划、卡池计划可回顾”。

关注点建议用自然语言写，并标记优先级。

| 关注点 | 优先级 | 说明 |
| --- | --- | --- |
| 今日应该完成什么 | 高 | 进入概览和任务功能块 |
| 哪些限时内容快结束 | 高 | 进入活动与版本、倒计时和概览 |
| 哪些内容值得长期回顾 | 中 | 写入时间轴 |
| 哪些资料需要安全保存 | 中 | 进入账号资产或资料功能块 |

方案后面的能力、功能块、字段、页面都应该能追溯到关注点。如果追溯不到，就先不要加。

## 9. 能力清单

能力是底层可复用机制。方案只引用能力，不重新发明能力。

第一阶段建议优先沉淀这些能力：

| 能力 | 解决的问题 |
| --- | --- |
| 目标系统 | 任务、清单、阶段目标、进度目标 |
| 周期与阶段 | 版本、赛季、旅行阶段、投保周期 |
| 活动与倒计时 | 限时活动、到期提醒、出发倒计时 |
| 资料资产 | 账号、证件、保单、链接、重要凭证 |
| 媒体记录 | 图片、截图、附件、票据 |
| 笔记知识 | 攻略、备注、经验、资料说明 |
| 预算金额 | 预算、支出、缴费、赔付 |
| 联系人 | 同行人、保险代理人、朋友、家庭成员 |
| 时间轴回顾 | 按天、周、阶段、版本回看发生过的事 |
| 摘要提醒 | 从功能块中生成概览重点 |
| AI 总结建议 | 生成建议、总结、风险提示、方案设计辅助 |
| 外部候选收件箱 | 接收飞书、Webhook、MCP 等外部候选内容 |

一个方案不需要启用所有能力。能力清单只说明这个方案需要哪些底层机制。

## 10. 功能块实例

功能块是用户可见入口，功能块实例是功能块在某个方案里的具体配置。

一份方案最重要的部分，就是功能块实例。它决定项目详情页有哪些 tab，概览能展示什么，管理页能配置什么，时间轴记录哪些内容。

功能块实例建议包含这些内容：

| 中文概念 | 推荐 key | 说明 |
| --- | --- | --- |
| 实例 ID | `instanceId` | 方案内稳定唯一值 |
| 功能块类型 | `blockKey` | 任务、活动、资料、图册、时间轴等 |
| 显示名称 | `displayName` | 当前方案里的叫法 |
| 使用能力 | `capabilities` | 这个功能块组合了哪些能力 |
| 是否启用 | `enabled` | 默认是否启用 |
| 是否进项目导航 | `navigation.visible` | 是否成为一级 tab |
| 导航顺序 | `navigation.order` | tab 排序 |
| 字段配置 | `fields` | 当前方案额外字段 |
| 行为规则 | `behavior` | 重置、提醒、状态流转、归档 |
| 概览规则 | `summaryRules` | 是否和如何进入概览 |
| 时间轴规则 | `timelineRules` | 哪些操作写入时间轴 |
| AI 规则 | `aiRules` | AI 可读、可总结、可建议范围 |
| 安全策略 | `security` | 敏感字段、展示脱敏、确认流程 |

判断一个功能块是否应该进入项目导航，可以用这个标准：

如果用户需要高频查看和维护完整数据，它应该进入项目导航。比如无限暖暖的任务、活动与版本、图册、账号资产。

如果它主要提供摘要、建议或底层支撑，可以不进入项目导航。比如无限暖暖第一阶段的素材收集和 AI 助手。

如果它主要管理规则，而不是管理日常数据，应放在项目管理页。比如摘要规则、时间轴写入规则、提醒策略。

## 11. 项目导航

项目导航由方案和功能块实例共同决定，不是固定页面集合。

项目详情页通常包含：

```text
概览
若干功能块 tab
时间轴
```

项目管理页不建议作为普通 tab 混在日常使用导航里。它是规则配置入口，应通过详情页右上角设置、项目列表操作或专门管理入口进入。

协议里建议显式声明导航：

```text
默认项目导航：
1. 概览
2. 任务
3. 活动与版本
4. 图册
5. 时间轴
6. 账号资产

隐藏但启用的能力：
素材收集、AI 助手、笔记攻略
```

这能避免所有能力都变成 tab，也能解释“功能块和 tab 不是一一对应关系”。

## 12. 概览摘要规则

概览不直接读取全部数据。概览读取摘要规则，由规则生成摘要。

每条摘要规则建议包含：

| 中文概念 | 推荐 key | 说明 |
| --- | --- | --- |
| 摘要 ID | `summaryId` | 稳定唯一值 |
| 来源功能块 | `sourceBlock` | 来自哪个功能块实例 |
| 摘要标题 | `title` | 用户看到的标题 |
| 是否启用 | `enabled` | 默认是否展示 |
| 优先级 | `priority` | 排序和异常优先级 |
| 最大展示数量 | `maxItems` | 防止概览变成列表页 |
| 筛选规则 | `filters` | 例如今日、本周、临近结束、异常 |
| 点击去向 | `targetTab` | 点击后进入哪个 tab |

概览摘要应该回答“现在最值得注意什么”，而不是“这个项目所有数据是什么”。

## 13. 时间轴规则

时间轴用于长期回顾，不是所有操作的流水账。

每个功能块实例都要声明哪些事件写入时间轴，以及写入粒度。

建议支持这些写入模式：

| 写入模式 | 说明 | 适用场景 |
| --- | --- | --- |
| 逐条记录 | 每次操作都写事件 | 活动归档、重要照片、保单变更 |
| 每日汇总 | 当天聚合成一条 | 日常任务、学习打卡 |
| 每周汇总 | 每周聚合成一条 | 周常目标、长期复盘 |
| 仅异常记录 | 只记录跳过、过期、风险 | 敏感资产、提醒失败 |
| 用户确认后记录 | AI 或外部内容先生成候选 | AI 总结、飞书导入 |
| 不记录 | 不进入时间轴 | 临时筛选、普通查看操作 |

长期经营型项目必须重视时间轴。阶段计划型项目也需要时间轴，但更常按阶段、日期或里程碑回顾。

## 14. 项目管理配置

项目管理页负责规则，不负责承接全部日常数据。

协议里要声明管理页配置项，通常分为三层。

项目基础配置包括项目名、封面、状态、周期属性、默认提醒渠道、敏感信息策略、AI 是否启用。

功能块实例配置包括启停、显示名、导航顺序、字段配置、摘要规则、提醒规则、归档规则、时间轴规则、AI 可读规则。

业务专项配置包括任务模板、行程阶段、预算分类、资料分组、联系人角色、图册同步、外部渠道映射。

管理页必须区分“方案默认配置”和“用户覆盖配置”。这会影响后续方案升级。

```text
方案默认值：无限暖暖任务每日 04:00 重置
用户覆盖值：用户把重置时间改成 05:00
方案升级时：不能直接覆盖用户的 05:00
```

## 15. 示例数据

方案可以带示例数据，用于演示和初始化。

示例数据要少而真实。它应该帮助用户理解方案，而不是伪造一整套复杂项目。

建议每个方案至少提供：

| 示例数据 | 用途 |
| --- | --- |
| 1 个项目基础信息 | 让项目卡片和详情页能展示 |
| 2-3 条概览摘要 | 验证概览规则 |
| 2-3 个功能块实例 | 验证导航和管理页 |
| 3-5 条核心业务数据 | 验证功能块 tab |
| 3-5 条时间轴事件 | 验证长期回顾 |

示例数据必须标记为 mock 或 seed，不能和真实用户数据混淆。

## 16. 安全与 AI 边界

方案必须声明安全和 AI 边界。

敏感字段默认不在首页、项目卡片、概览摘要和普通列表中明文展示。账号、证件、保单、联系人、支付凭证、Token 都属于敏感内容。

AI 可以读取哪些内容，必须由功能块实例声明。AI 默认可以读取非敏感摘要、状态、时间轴公开事件；涉及账号、证件、联系人、保单细节时，需要用户确认。

AI 可以生成建议和候选内容，但不应无确认写入关键资料。外部渠道导入内容也一样，应先进入待确认或候选状态。

## 17. 方案升级与继承

方案会升级，用户项目也会个性化。因此协议需要保留继承关系。

建议区分三层：

```text
方案默认配置
  -> 用户创建项目时复制为项目配置
      -> 用户后续覆盖部分配置
```

方案升级时，只能自动更新用户没有覆盖过的配置。用户覆盖过的配置，应提示用户选择是否合并。

每个可覆盖配置都应记录来源：

| 来源 | 含义 |
| --- | --- |
| `scheme_default` | 来自方案默认值 |
| `user_override` | 用户在项目中改过 |
| `ai_suggested` | AI 建议，等待确认或已确认 |
| `imported` | 外部渠道导入 |

## 18. 推荐数据结构草案

下面是第一阶段可用于 mock 和前端配置的结构草案。它不是最终后端表结构，但后续表结构和接口可以从这里演进。

```ts
interface SchemeTemplate {
  schemeId: string;
  schemeName: string;
  type: {
    key: string;
    name: string;
  };
  cycleType: 'long_running' | 'stage_planned' | 'hybrid';
  version: string;
  status: 'draft' | 'available' | 'archived';
  source: 'official' | 'user' | 'community' | 'ai_generated';
  summary: string;
  targetUser: string;
  usageFrequency: string;
  complexity: 'light' | 'standard' | 'advanced';
  focusPoints: SchemeFocusPoint[];
  capabilities: SchemeCapabilityRef[];
  blockInstances: SchemeBlockInstance[];
  navigation: SchemeNavigationItem[];
  overviewRules: SchemeOverviewRule[];
  managementSections: SchemeManagementSection[];
  timelineRules: SchemeTimelineRule[];
  seedData: SchemeSeedData;
  safety: SchemeSafetyPolicy;
}
```

功能块实例结构建议：

```ts
interface SchemeBlockInstance {
  instanceId: string;
  blockKey: string;
  displayName: string;
  enabled: boolean;
  capabilities: string[];
  navigation: {
    visible: boolean;
    order: number;
  };
  fields: SchemeFieldConfig[];
  behavior: {
    resetRules?: unknown[];
    reminderRules?: unknown[];
    archiveRule?: unknown;
    statusFlow?: string[];
  };
  summaryRules: SchemeOverviewRule[];
  timelineRules: SchemeTimelineRule[];
  aiRules: {
    readable: boolean;
    writableAfterConfirm: boolean;
    allowedUse: string[];
  };
  security: {
    sensitivity: 'normal' | 'private' | 'sensitive';
    maskInOverview: boolean;
    requireConfirmBeforeExternalWrite: boolean;
  };
}
```

## 19. 示例：无限暖暖方案片段

```text
方案名称：无限暖暖方案
所属类型：游戏
周期属性：长期经营型
目标用户：长期游玩无限暖暖、希望管理日常、活动、素材、图册和账号资料的玩家
关注点：今日任务、本周目标、版本活动、活动倒计时、素材收集、图册记录、账号资产、长期时间轴
```

默认功能块实例：

| 功能块实例 | 是否进导航 | 使用能力 | 说明 |
| --- | --- | --- | --- |
| 概览 | 是 | 摘要提醒、AI 建议 | 展示今日、本周、活动、素材、资产、时间轴摘要 |
| 任务 | 是 | 目标系统、周期规则、时间轴 | 管理日常、周常、活动目标、自定义目标 |
| 活动与版本 | 是 | 周期与版本、活动管理、倒计时、目标系统 | 管理版本、活动、倒计时、活动归档 |
| 图册 | 是 | 媒体记录、活动归属、时间轴 | 管理截图和搭配照片 |
| 时间轴 | 是 | 时间轴回顾、AI 总结 | 按天、周、版本回顾 |
| 账号资产 | 是 | 资料资产、安全策略 | 管理账号、UID、兑换码、链接 |
| 素材收集 | 否 | 收集与养成、摘要提醒 | 第一阶段进入概览和活动归档，不做独立 tab |
| AI 助手 | 否 | AI 总结建议 | 进入概览建议、时间轴总结和管理配置 |

这个例子说明：同样是启用的能力，不一定都成为 tab。

## 20. 示例：日本旅行方案片段

```text
方案名称：日本自由行方案
所属类型：旅行
周期属性：阶段计划型
目标用户：正在规划一次日本旅行的人
关注点：出发倒计时、行程、机票酒店、证件保险、行前清单、预算、天气、同行人、照片和旅行后回顾
```

默认功能块实例：

| 功能块实例 | 是否进导航 | 使用能力 | 说明 |
| --- | --- | --- | --- |
| 概览 | 是 | 摘要提醒、倒计时、AI 建议 | 展示出发倒计时、近期行程、清单进度、风险提醒 |
| 行程 | 是 | 周期与阶段、地点、提醒、时间轴 | 管理每日安排、地点、交通和预订 |
| 清单 | 是 | 目标系统、提醒 | 管理行前准备、随身物品、待购买 |
| 资料 | 是 | 资料资产、安全策略 | 管理护照、签证、机票、酒店、保险 |

这个例子说明：阶段计划型项目更强调准备期、执行期和完成归档，而不是每日长期循环。

## 21. AI 生成新方案时的输出格式

未来 AI 帮用户设计方案时，应按这个顺序输出：

```text
1. 方案名称和所属类型
2. 周期属性判断
3. 适用人群和不适用场景
4. 用户关注点
5. 需要的底层能力
6. 默认功能块实例
7. 项目导航和隐藏能力
8. 概览摘要规则
9. 时间轴记录规则
10. 项目管理配置项
11. 示例数据建议
12. 敏感信息和 AI 边界
13. 仍需用户确认的问题
```

AI 不应直接从字段开始，也不应默认把每个能力都变成 tab。遇到不确定的地方，要标记为待确认，而不是用想当然的字段补满。

## 22. 验收标准

一份方案模板通过验收，应该能做到：

用户能根据方案创建一个项目。

系统能根据方案生成项目详情页导航。

概览能根据摘要规则展示重点，而不是写死页面内容。

功能块 tab 能承接日常数据新增、查看、编辑和归档。

项目管理页能配置功能块实例，而不是只改项目基础信息。

时间轴能按项目周期属性展示长期或阶段回顾。

AI 能按方案理解项目语境，给出建议，但不会无确认写入敏感资料。

开发能根据方案继续落 mock 数据、接口和后端表结构。
