# Life Manager 无限暖暖项目设计草案

## 1. 文档定位

本文只围绕“无限暖暖”这个长期游戏项目展开，但目标不是只设计一个无限暖暖页面。

更准确地说，本文用无限暖暖作为样本，先思考长期游戏项目底层到底要管理什么，再抽象成“游戏类型通用能力”，最后说明无限暖暖方案如何选择、命名和配置这些能力。

如果要进入前端 mock、页面改造和接口演进，请继续参考 [Life Manager 无限暖暖数据模型与页面闭环设计.md](./Life%20Manager%20无限暖暖数据模型与页面闭环设计.md)。那份文档会把版本、活动、目标、概览摘要、配置和时间轴串成可开发规格。

后续设计原神、三角洲行动、明日方舟或其他游戏时，不应该直接复制无限暖暖页面，而应该复用这里沉淀出来的底层能力，再由各自方案做差异化组合。

## 2. 推导顺序

这类项目不能从“页面有哪些字段”开始，也不能只从“任务功能块怎么显示”开始。

正确顺序应该是：

```text
长期游戏项目底层需求
  -> 游戏类型通用能力
  -> 具体游戏方案
  -> 用户个性化功能块
  -> 页面与交互
```

其中，游戏类型通用能力回答“长期玩一个游戏时，系统应该能管理哪些抽象对象”；无限暖暖方案回答“无限暖暖默认启用哪些能力、怎么命名、怎么展示、怎么配置”；用户个性化功能块回答“用户自己的玩法如何加入，而不污染通用模型”。

## 3. 长期游戏项目底层需要什么

长期游戏项目的核心不是“每天打卡”，而是一个持续变化的游戏生活空间。它会经历版本、活动、日常循环、阶段目标、资源积累、截图记录、资料变化和回顾总结。

第一类底层需求是时间结构。长期游戏虽然没有明确结束点，但它不是一条无限长的时间线，而是由版本、赛季、活动、每日重置、每周重置、限时节点组成。系统必须能知道当前处在哪个版本、哪个活动快结束、哪些任务随周期刷新、哪些内容应该在版本结束后归档。

第二类底层需求是目标结构。用户每天要做什么、本周要推进什么、某个活动要完成什么、自己想收集什么、想拍什么、想培养什么，这些都属于目标。但目标不一定都叫任务。日常是周期目标，活动任务是限时目标，素材收集是收集目标，拍照计划是创作目标，角色培养或装备配置是养成目标。

第三类底层需求是内容归属。一个任务、一张截图、一篇攻略笔记、一个素材目标，常常不只是属于项目本身，还属于某个版本、某个活动、某个套装目标或某个自定义计划。没有内容归属，时间久了以后所有东西都会堆成一团。

第四类底层需求是资料和资产。长期游戏项目会保存账号资料、服务器、UID、充值记录、兑换码、攻略链接、素材来源、截图目录、外部工具链接。这些内容不一定每天使用，但需要安全、可查、可关联。

第五类底层需求是回顾。长期项目的价值不只是提醒今天做什么，还要能回答“我这段时间玩了什么”“这个版本做了什么”“哪些活动错过了”“留下了哪些截图和笔记”。因此时间轴不是装饰，而是游戏项目的底层能力。

第六类底层需求是注意力管理。长期游戏信息很多，概览页不应该把所有内容摊开，而是帮用户判断现在最该关注什么。这个判断来自摘要规则、提醒规则、置顶规则和异常规则。

## 4. 底层对象模型

为了支撑上面的需求，游戏类型至少需要这些底层对象。

| 对象 | 含义 | 无限暖暖中的例子 |
| --- | --- | --- |
| 游戏项目 | 用户长期维护的游戏空间 | 我的无限暖暖 |
| 游戏周期 | 承载版本、赛季、阶段 | 1.5 版本、当前版本 |
| 游戏活动 | 有开始和结束时间的内容容器 | 限时活动、版本活动 |
| 目标 | 用户要完成或推进的事 | 每日任务、周常、活动任务、拍照计划 |
| 收集项 | 需要积累、兑换、补齐的对象 | 素材、套装、兑换道具 |
| 资料资产 | 稳定保存的信息 | 账号、UID、兑换码、攻略链接 |
| 媒体记录 | 图片、截图、视频、附件 | 搭配截图、活动截图 |
| 笔记攻略 | 用户或外部来源的知识 | 活动攻略、搭配备注 |
| 时间轴事件 | 项目历史中发生过的事 | 完成任务、活动归档、上传截图 |
| 能力实例 | 某个能力在项目里的配置结果 | 无限暖暖的“日常任务”实例 |

这些对象之间的关系比字段更重要。

```text
游戏项目
  -> 游戏周期
      -> 游戏活动
          -> 活动目标
          -> 活动倒计时
          -> 活动截图
          -> 活动笔记

游戏项目
  -> 周期目标
      -> 日常
      -> 周常

游戏项目
  -> 收集项
  -> 资料资产
  -> 媒体记录
  -> 时间轴事件
```

这样建模以后，版本限时活动就不会被误塞进“普通任务列表”。它应该先是一个活动对象，活动下面可以有任务、倒计时、素材、截图和笔记。

## 5. 游戏类型通用能力

从底层对象可以抽象出一组游戏类型通用能力。它们不是页面，也不是某个游戏的字段，而是未来所有游戏方案都可以复用的能力。

| 通用能力 | 解决的问题 | 可形成的视图 |
| --- | --- | --- |
| 周期与版本 | 游戏内容随版本、赛季、周期变化 | 当前版本、版本归档、赛季进度 |
| 活动管理 | 限时内容有时间范围和关联内容 | 活动列表、活动详情、活动倒计时 |
| 目标系统 | 日常、周常、活动目标、自定义目标 | 今日、本周、活动任务、目标看板 |
| 收集与养成 | 管理素材、套装、角色、装备等长期目标 | 素材、套装、养成计划 |
| 资料资产 | 保存账号、链接、兑换码、攻略资料 | 账号资料、攻略库、重要链接 |
| 媒体记录 | 保存截图、照片、视频和附件 | 图册、活动照片、作品集 |
| 笔记攻略 | 沉淀玩法理解和外部资料 | 攻略笔记、活动备注 |
| 时间轴回顾 | 回看每天、每周、每个版本发生了什么 | 日回顾、周回顾、版本总结 |
| 摘要提醒 | 从大量内容中提取当前重点 | 概览、提醒、异常提示 |
| 配置系统 | 控制能力如何展示、记录和归档 | 项目配置、能力实例配置 |

这些通用能力可以被不同游戏方案重新命名和组合。

无限暖暖会重视素材、搭配、拍照、活动；原神会重视树脂、角色养成、深渊、卡池；三角洲行动可能重视赛季、通行证、武器配置、战绩复盘。它们不应该共享同一个固定页面，但应该共享“周期、活动、目标、收集、资料、媒体、时间轴、配置”这些底层能力。

## 6. 无限暖暖方案如何使用通用能力

无限暖暖方案不是重新发明一套能力，而是对游戏类型通用能力进行选择、改名和默认配置。

| 游戏通用能力 | 无限暖暖方案中的名称 | 默认作用 |
| --- | --- | --- |
| 周期与版本 | 版本 | 管理当前版本、版本节点、版本归档 |
| 活动管理 | 活动与版本 | 管理限时活动、活动倒计时、活动归档 |
| 目标系统 | 任务 | 管理日常、周常、活动任务、自定义目标 |
| 收集与养成 | 素材收集 | 管理素材、套装、搭配目标、拍照计划 |
| 资料资产 | 账号资产 | 管理账号、UID、兑换码、攻略链接 |
| 媒体记录 | 图册 | 管理截图、搭配照片、活动照片 |
| 笔记攻略 | 时间轴记录 | 管理攻略、搭配备注、活动记录，不作为第一阶段独立 tab |
| 时间轴回顾 | 时间轴 | 按天、周、版本回顾项目历史 |
| 摘要提醒 | 概览 | 展示当前最需要关注的重点 |
| 配置系统 | 项目管理 | 配置每个能力实例的显示和行为 |

这里要特别注意：无限暖暖的“任务”不是一个孤立功能块，它经常需要和“活动与版本”联动。版本活动任务应该属于任务体系，但它的时间范围、倒计时和归档规则来自活动对象。

## 7. 无限暖暖项目第一阶段导航

第一阶段不把每个能力都做成一级 tab。tab 只承载用户最常进入、需要完整查看和管理数据的功能块；其余能力进入概览摘要、时间轴和项目管理页。

当前项目详情页默认导航为：

```text
概览
任务
活动与版本
图册
时间轴
账号资产
```

概览只回答“现在最该关注什么”。它展示今日摘要、本周摘要、当前版本摘要、即将结束活动、最近时间轴和 AI 建议。它不展示全部任务，也不负责完整管理。

任务页负责完整目标系统。日常、周常、活动任务、自定义目标都在这里维护，但必须能按类型、版本、活动、状态和是否归档筛选。

活动与版本页负责版本和活动，也承担第一阶段的“进行中上下文”。它管理当前版本、活动起止时间、活动倒计时、活动关联任务、活动素材、活动截图和活动归档。

图册页负责截图、搭配照片和活动照片。照片可以关联版本、活动和目标，上传或整理后可以按配置写入时间轴。

账号资产页负责账号、UID、兑换码、支付凭证、外部链接等稳定资料。敏感内容默认不进入概览，时间轴也只记录“发生过变更”，不展示明文。

时间轴页负责长期回顾。它按天、按周、按版本展示项目历史，不应该只是最近动态列表。

素材收集、笔记攻略和 AI 助手第一阶段不作为默认一级 tab。素材收集进入概览摘要、活动归档和 AI 复盘；笔记攻略由 `NikkiNote` 承载，入口先放在概览摘要、活动详情和时间轴中，后续可升级为独立 tab；AI 助手进入概览建议、时间轴总结和项目管理配置。

项目管理页不属于详情页 tab。它负责项目基础配置和功能块实例配置，包括启停、导航显示、概览摘要规则、时间轴写入规则、提醒规则和归档规则。

## 8. 任务能力的底层拆分

任务能力应该抽象成“目标系统”，而不是简单待办。它至少支持四类目标。

日常目标按日重置，用来承载每天要做的事。它适合进入概览，但只显示摘要和少数重点。

周常目标按周重置，用来承载一周内推进的事。它适合在概览中显示进度，不适合展开完整列表。

活动目标挂在活动对象下面，用来承载版本或活动期间要完成的事。它继承活动的开始时间、结束时间、倒计时和归档规则。

自定义目标由用户自己创建，可以是一次性目标、重复目标、收集目标或创作目标。它是否进入概览、是否写入时间轴、是否归档，都应由用户自己配置。

| 目标类型 | 上层归属 | 时间规则 | 概览显示 | 归档规则 |
| --- | --- | --- | --- | --- |
| 日常目标 | 项目 | 每日重置 | 今日进度和异常项 | 每日汇总 |
| 周常目标 | 项目 | 每周重置 | 本周进度 | 周总结 |
| 活动目标 | 活动 | 继承活动时间 | 临近结束或置顶活动 | 活动或版本归档 |
| 自定义目标 | 用户指定 | 用户指定 | 用户决定 | 用户决定 |

这意味着“版本限时活动任务”不是和日常、周常平铺在一起的普通任务。它是活动对象下的目标，任务页可以查看它，但活动与版本页才是它最自然的上下文。

## 9. 版本与活动能力

版本与活动是长期游戏项目的核心能力之一。

版本用于承载一段游戏内容周期。它有开始时间、结束时间、状态、版本重点、关联活动、版本总结和归档状态。

活动用于承载一个限时内容。它有开始时间、结束时间、倒计时规则、关联目标、关联素材、关联截图、关联笔记和归档结果。

```text
版本
  名称
  开始时间
  结束时间
  状态：未开始 / 进行中 / 即将结束 / 已结束 / 已归档
  版本重点
  关联活动
  版本总结

活动
  所属版本
  名称
  开始时间
  结束时间
  状态：未开始 / 进行中 / 即将结束 / 已结束 / 已结束待归档 / 已归档
  倒计时规则
  关联目标
  关联素材
  关联截图
  关联笔记
```

版本结束后，系统应提示归档。归档不是删除，而是把版本内的活动、目标完成情况、截图、笔记、素材进展和 AI 总结沉淀到时间轴。新版本开始时，创建新的版本对象和活动对象，旧版本内容不再混在当前进行中的任务里。

## 10. 概览如何从底层能力生成

概览不是功能块集合，而是底层能力生成的摘要。

无限暖暖概览可以由这些摘要组成：

```text
今日摘要：今天剩几项，是否有高优先级未完成
本周摘要：周常进度、遗漏风险
版本摘要：当前版本剩余时间、版本重点
活动摘要：即将结束活动、未完成活动目标
素材摘要：当前最重要的素材缺口或套装目标
图册摘要：最近截图、置顶照片
时间轴摘要：最近几天发生了什么
AI 建议：根据当前版本、任务和时间轴给出建议
```

每个摘要都应该有配置，而不是写死。

```text
显示哪些摘要
每个摘要最多展示几条
哪些异常优先展示
哪些自定义目标可以置顶
哪些活动进入概览
哪些内容只在对应详情 tab 展示
```

这样，日常、周常、版本活动不会被迫挤在一个大任务卡片里。概览只展示它们各自产出的摘要，用户点击后进入对应上下文。

## 11. 用户个性化功能块

用户长期玩下去以后，一定会产生方案外的玩法。系统不能要求所有个性化内容都变成全局功能块。

用户可以增加三类个性化能力。

第一类是基于已有通用能力创建新实例。例如用目标系统创建“小号维护”，用资料资产创建“氪金记录”，用提醒能力创建“直播提醒”。

第二类是给已有能力增加字段或分组。例如在素材搭配中增加“拍照地点”，在图册中增加“构图标签”，在任务中增加“小号”分组。

第三类是创建只属于当前项目的自定义功能块。例如“朋友互助”“拍照挑战”“每周复盘”。这类内容先留在当前项目，不急着抽象到游戏类型。

新增时，系统应让用户回答一个简单问题：

```text
这是已有能力的新分组，还是一个新的能力实例？
它是否需要进入项目导航？
它是否需要出现在概览？
它是否需要写入时间轴？
它是否需要跟版本或活动关联？
```

## 12. 项目配置应该配置什么

配置页要分成两层。

项目基础配置负责项目名、封面、状态、账号安全级别、当前版本、默认提醒渠道、AI 是否启用、敏感信息策略。

能力实例配置负责每个能力在无限暖暖项目里的表现。以目标系统为例，它需要配置：

```text
显示名称：任务 / 日常 / 暖暖目标
启用目标类型：日常、周常、活动、自定义
重置规则：每日几点、每周哪天、活动结束后如何处理
概览摘要：显示今日、周常、活动、异常还是自定义置顶
活动关联：活动目标是否必须绑定活动
倒计时来源：目标自己截止时间，还是继承活动结束时间
时间轴记录：逐条记录、每日汇总、仅记录异常、完全不记录
归档规则：完成后保留多久，版本结束后是否自动归档
AI 规则：是否允许 AI 根据目标记录生成建议和总结
```

活动与版本能力也需要配置：

```text
是否启用版本周期
版本结束前几天提醒
活动结束前几天提醒
版本结束后是否自动生成回顾
归档时是否关联目标、截图、笔记、素材
是否允许 AI 从外部内容创建候选活动
```

图册、资料、素材搭配、时间轴都应该有自己的实例配置。否则用户无法控制它们在概览中如何出现、哪些操作写入时间轴、哪些内容归档。

## 13. 时间轴与长期回顾

时间轴是长期游戏项目的底层能力。它应该支持按天、按周、按版本回顾。

按天回顾时，用户可以看到今天完成了哪些日常，哪些目标跳过了，上传了哪些截图，新增了哪些素材目标，AI 给了什么建议。

按周回顾时，用户可以看到周常完成情况、连续完成天数、遗漏次数、截图数量、素材目标推进情况。

按版本回顾时，用户可以看到这个版本参与了哪些活动，完成了哪些活动目标，错过了哪些内容，留下了哪些截图和笔记。

时间轴事件来源可以包括：

```text
目标完成
目标跳过
周常遗漏
活动开始
活动即将结束
活动结束
版本归档
素材目标完成
截图上传
笔记新增
资料更新
AI 生成总结
```

但时间轴不能把所有操作都记录成噪音。每个能力实例都要能配置记录粒度。

```text
日常目标：每日汇总
周常遗漏：逐条记录
活动目标：完成和逾期逐条记录
图册上传：每日汇总，置顶照片逐条记录
资料变更：敏感资料只记录“有变更”，不显示明文
AI 总结：用户确认后写入
```

## 14. 最终抽象

从无限暖暖可以抽象出一套游戏类型通用能力：

```text
游戏项目
  周期与版本能力
  活动管理能力
  目标系统能力
  收集与养成能力
  资料资产能力
  媒体记录能力
  笔记攻略能力
  时间轴回顾能力
  摘要提醒能力
  能力实例配置
```

具体游戏方案不应该重新设计底层能力，而应该选择这些能力、改名、排序、配置默认规则，并补充该游戏特有的字段和示例数据。

无限暖暖方案重点配置活动、搭配、素材、拍照和图册；原神方案可以重点配置资源恢复、角色养成、深渊和卡池；三角洲行动方案可以重点配置赛季、通行证、装备配置和战绩复盘。

这样，Life Manager 以后就不是为每个游戏重写一套页面，而是拥有一个稳定的游戏项目底座，再用方案表达不同游戏的关注点。

## 15. 当前结论

无限暖暖项目的底层重点不是“任务功能块”，而是长期游戏项目如何组织时间、活动、目标、收集、记录和回顾。

版本限时活动应该属于目标系统，但它的上层归属是活动。它继承活动倒计时和版本归档规则，因此不能简单平铺在普通任务列表里。

概览页应该展示摘要，不展示全部内容。摘要来自底层能力和能力实例配置。

项目配置页必须能进入每个能力实例配置。否则用户无法控制概览展示什么、目标如何重置、活动如何归档、时间轴如何记录。

最终可以抽象出“游戏类型通用能力”，再由无限暖暖、原神、三角洲行动等方案去组合和配置。这个方向比直接罗列页面字段更稳。
